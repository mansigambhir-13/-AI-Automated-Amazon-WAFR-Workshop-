**Create `wafr-agents/scripts/entrypoint.sh`:**

```bash
#!/bin/bash
set -e

echo "=== Running file-to-DynamoDB migration (idempotent, safe to re-run) ==="
cd /app
python3 scripts/migrate_sessions.py \
  --sessions-dir review_sessions/sessions \
  --pipeline-dir review_sessions/pipeline_results \
  || echo "WARNING: Migration exited with errors — check logs above. Server will still start."

echo "=== Migration complete. Starting uvicorn server... ==="
exec uvicorn wafr.ag_ui.server:app \
  --host 0.0.0.0 \
  --port 8000 \
  --timeout-keep-alive 300 \
  --workers 1 \
  --log-level info
```

Key design choices:
- `set -e` but migration failure is caught with `||` — migration errors should NOT prevent the server from starting (the server can operate on DynamoDB even if some legacy files failed to migrate)
- `exec` replaces the shell process with uvicorn so signals (SIGTERM from App Runner) reach the server directly
- Uses `python3` (not `python`) per the issue noted in 02-03 SUMMARY
- `cd /app` ensures correct WORKDIR even if entrypoint is invoked from elsewhere

**Update `wafr-agents/Dockerfile`:**

1. Add `COPY scripts/ ./scripts/` AFTER the existing `COPY __init__.py .` line — this brings the migration script and entrypoint into the image

2. Add `COPY review_sessions/ ./review_sessions/` BEFORE the existing `RUN mkdir -p review_sessions/...` line — this copies the actual session JSON data files into the image for the one-time migration. The `mkdir -p` line can remain as a safety fallback.

3. Add `RUN chmod +x scripts/entrypoint.sh` after the COPY scripts line

4. Replace the existing multi-line CMD:
```dockerfile
CMD ["uvicorn", "wafr.ag_ui.server:app", \
     "--host", "0.0.0.0", \
     "--port", "8000", \
     "--timeout-keep-alive", "300", \
     "--workers", "1", \
     "--log-level", "info"]
```
With:
```dockerfile
CMD ["bash", "scripts/entrypoint.sh"]
```

The final Dockerfile should look like:
```dockerfile
FROM python:3.11-slim

ENV PYTHONDONTWRITEBYTECODE=1
ENV PYTHONUNBUFFERED=1
ENV REVIEW_STORAGE_TYPE=dynamodb

WORKDIR /app

RUN apt-get update && apt-get install -y --no-install-recommends \
    gcc \
    && rm -rf /var/lib/apt/lists/*

COPY requirements.txt .
RUN pip install --no-cache-dir --upgrade pip && \
    pip install --no-cache-dir -r requirements.txt && \
    pip install --no-cache-dir "uvicorn[standard]" "fastapi>=0.109.0"

COPY wafr/ ./wafr/
COPY knowledge_base/ ./knowledge_base/
COPY __init__.py .
COPY scripts/ ./scripts/
RUN chmod +x scripts/entrypoint.sh
COPY review_sessions/ ./review_sessions/
RUN mkdir -p review_sessions/pipeline_results review_sessions/sessions /tmp/reports

EXPOSE 8000

HEALTHCHECK --interval=30s --timeout=5s --start-period=15s --retries=3 \
    CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:8000/health')" || exit 1

CMD ["bash", "scripts/entrypoint.sh"]
```

Note: The `COPY review_sessions/ ./review_sessions/` copies the 10 session files and 4 pipeline result files into the image. This is safe — these are assessment data (not secrets). After migration runs once and all items are in DynamoDB, subsequent startups will complete the migration check in less than 1 second (all items skipped as already present).
